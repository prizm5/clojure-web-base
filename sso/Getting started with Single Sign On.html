Getting started with Single Sign On

Last Updated: 11/6/2014
RELATED LINKS
  • OAuth 2.0 specification
  • OAuth 2.0 Threat Models
  • IBM® Bluemix™ Pricing Sheet
  • IBM Bluemix Prerequisites

IBM Single Sign On for Bluemix is a policy-based authentication service that provides an easy to embed single sign-on capability for web applications. The service supports an OAuth 2.0 authorization code flow with a user profile as a protected API resource.

A sample application is provided at https://hub.jazz.net/project/bluemixsso/SingleSignOnSampleClient/overview. The sample application can help you write your own client application to leverage the IBM Single Sign On for Bluemix service's capabilities. If you'd like, you can fork the project to your own IBM DevOps project. Use the instructions provided within the project to obtain the source code, build it, and then deploy it to your own IBM Bluemix space. For your reference, an instance of the sample code has been deployed to the IBM Bluemix platform, and is available at https://idaastestclient.mybluemix.net/.

Single Sign On with OAuth 2.0

An application may use an OAuth 2.0 flow for requesting access to end-user profile data via an API. Using this method an application can determine the identity of the current user and use that data to effect a login. This is the same way, for example, that authentication can be achieved using the Facebook Graph API.

Here is an end-to-end diagram that shows how the OAuth 2.0 authorization code flow works for obtaining user profile information with the various actors involved:

  1. User is interacting with the application and the application decides it needs a grant from the user (resource owner) to access the "profile" API.
  2. User is redirected to the authorization endpoint of the OAuth server, requesting the "profile" scope. User is required to authenticate and grant access.
  3. User is redirected back to application with authorization code.
  4. Application, acting as an OAuth client, exchanges authorization code for an access token.
  5. Application calls profile API with the access token. Profile resource validates the access token, then returns the user profile corresponding to the user who granted access. The application can then use this information to establish an authenticated session with the user.

How the Single Sign On service works

The Single Sign On Service flow consists of the following steps:

  1. Redirection for authentication and authorization
  2. Implementation of a redirect_uri
  3. Retrieval of the profile resource

1. Redirecting for authentication and authorization

When your application requires user authentication, it redirects the user's browser to the authorize endpoint of the single sign-on service. This redirect contains a set of query string parameters. Here is an example redirect URL:

https://idaas.ng.bluemix.net/sps/oauth20sp/oauth20/authorize?client_id=mGELGfzaM6obYnQedoJt&response_type=code&scope=profile&redirect_uri=https://idaastestclient.mybluemix.net/index.jsp&state=2663620a-4b4d-45f4-82b6-20439940887b&requestedAuthnPolicy=http://www.ibm.com/idaas/authnpolicy/basic
Table 1. Query string parameters
Parameter Description
client_id Assigned when you bind the Single Sign On service to your application in Bluemix. See "Creating a Single Sign On client configuration" for more details.
response_type Must be "code" for the authorization code grant type as defined in the OAuth 2.0 specification.
scope Must be "profile" to request the user's profile.
redirect_uri Must be an https URL of your application, to which the user's browser is redirected after authentication. This URL also needs to be configured as part of your application binding.

See Creating a Single Sign On client configuration for more details.
state Defined by your application and verified at the redirect_uri to prevent cross-site-request-forgery attacks against your redirect_uri. Optional, but typically used.
requestedAuthnPolicy Instructs the Single Sign On service how to authenticate the user. Use is optional but if sent, it must be verified when single signon completes and the profile is read. This verification ensures that the Single Sign On service received and processed the same policy that your application requested.

When requestedAuthnPolicy is not provided in the query string, there is no specific policy in the request. The service behavior is the same as when requestedAuthnPolicy is provided with a value of standard.

When requestedAuthnPolicy is provided in the query string, it contains one of the following values:

http://www.ibm.com/idaas/authnpolicy/basic
Permit authentication at any level. If a user is already logged in at the Single Sign On service, no further authentication of that user is required.
http://www.ibm.com/idaas/authnpolicy/basic/verified_email
Permit authentication at any level. If the user already has a verified email address associated with their account, no further authentication of that user is required. If there is no verified email, the user is directed through a one-time-password process to verify it.
http://www.ibm.com/idaas/authnpolicy/standard
Requires the user to either use a "registered browser", or step-up to one-time-password (OTP) authentication. If the user is not using a registered browser, they can register their browser as part of that process.
http://www.ibm.com/idaas/authnpolicy/otp
Requires that the user's session at the Single Sign On service is established with OTP.
http://www.ibm.com/idaas/authnpolicy/reauth
Requires the user to re-establish their session at the Single Sign On service as part of this single sign-on flow. Any level of authentication is accepted. For identity providers that have long-lived browser sessions, such as social sites, the policy appears to have no effect. This effect occurs because the Single Sign On service has no effect on the browser session with the user's real identity provider site.
http://www.ibm.com/idaas/authnpolicy/reauth-otp
Requires the user to reauthenticate their session at the Single Sign On service as part of this single sign-on flow by using one-time password.

2. Implementing a redirect_uri

Your redirect URI must be an endpoint that can receive query string parameters. For example:

https://idaastestclient.mybluemix.net/index.jsp?code=kceVR6tJWYqkOXqNrJul37abyreaCU&state=2663620a-4b4d-45f4-82b6-20439940887b

The state parameter must match the value that your application sent in the redirect for authorization.

The code parameter must be used by your application to retrieve an access token, by using direct communications with the token endpoint of the Single Sign On service. The following curl command shows an example of exchanging the authorization code for an access token. 

curl -d "client_id=mGELGfzaM6obYnQedoJt&client_secret=REDACTED&grant_type=authorization_code&code=kceVR6tJWYqkOXqNrJul37abyreaCU&redirect_uri=https://idaastestclient.mybluemix.net/index.jsp" https://idaas.ng.bluemix.net/sps/oauth20sp/oauth20/token
Table 2. Curl command parameters
Parameter Description
client_id Must be the same client_id that was used in the redirect to the authorize URL.
client_secret Assigned when you bind the Single Sign On service to your application in Bluemix.

See Creating a Single Sign On client configuration for more details.
grant_type Must be "authorization_code" as defined in the OAuth 2.0 specification.
code The value of the code parameter that is received at the redirect_uri
redirect_uri The URL to which the code was sent

A successful exchange of an authorization code for an access token results in a JSON response. For example:

{"expires_in":3599,"scope":"profile","access_token":"a4qIr9JvSIA510KIi7LN","token_type":"bearer","refresh_token":"PcpGqTx58vh2fw1Y1aJEw4nrsvfZvAOe2I4C8sbX"}

3. Retrieving the profile resource

The access token can now be used to retrieve the user's profile. The response from the profile.jsp endpoint provides specifics on who the user is, so that your application can establish an authentication session. The following curl command demonstrates retrieval of the profile information:

curl -H "Authorization: bearer a4qIr9JvSIA510KIi7LN" https://idaas.ng.bluemix.net/idaas/resources/profile.jsp

Here is an example of a successful response:

{"lastName":["SMITH"],"userDisplayName":["jsmith@example.com"],"userRealm":"www.ibm.com","username":"http:\/\/www.ibm.com\/jsmith@example.com","userUniqueID":["http:\/\/www.ibm.com\/110000R11X"],"AZN_CRED_CREATE_TIME":"2014-10-07T01:15:21Z","requestedAuthnPolicy":"http:\/\/www.ibm.com\/idaas\/authnpolicy\/basic","idaas.verified_email":["jsmith@example.com"],"firstName":["JOHN"],"name":["JOHN SMITH"],"email":["jsmith@example.com"],"AUTHENTICATION_LEVEL":"2"}

Your application must verify that the value of requestedAuthnPolicy in the response matches the requestedAuthnPolicy sent in the initial redirect URL.

For information on all attributes in the profile, see User profile attributes.

Creating a Single Sign On client configuration

To use single sign-on authentication with your application, create a Single Sign On client configuration. You create an OAuth client configuration when you bind an instance of the Single Sign On service to your Bluemix application.

  1. After you bind the Single Sign On service instance to the application, click the Single Sign On service instance to create the client configuration.
  2. Select one or more identity providers. IBM is preselected.
  3. Select the OAuth 2.0 authentication protocol.
  4. In the Display Name field, type the name you want users to see as the application name on consent-to-authenticate forms.
  5. Select the Enabled check box to enable the configuration. Clear the check box if you want to disable the configuration.
  6. Enter the Redirect URI for your application where the OAuth authorization code will be sent after a user signs in.
  7. Click Save. The client secret and client identifier are generated and displayed. The configuration is displayed in read-only mode. Click Edit to make changes.

User profile attributes

The profile returned to your application is a JSON object that contains a number of attributes. Here are two example profiles, followed by an explanation of each attribute.

Here is an example profile that is built when a user authenticates at the Single Sign On service by using the Facebook Identity Provider (IDP):

{
    "lastName": ["Smith"],
    "username": "http:\/\/www.facebook.com\/1234567890",
    "userUniqueID": ["http:\/\/www.facebook.com\/1234567890"],
    "AUTHENTICATION_LEVEL": "2",
    "email": ["johnsmith@somwhere.com"],
    "name": ["John Smith"],
    "AZN_CRED_CREATE_TIME": "2014-09-24T02:27:26Z",
    "userRealm": "www.facebook.com",
    "userDisplayName": ["johnsmith@somewhere.com"],
    "firstName": ["John"],
    "requestedAuthnPolicy": "http:\/\/www.ibm.com\/idaas\/authnpolicy\/basic\/verified_email",
    "idaas.verified_email": ["johnsmith@somewhere.com"]
}

Here is an example profile that is built when a user authenticates at the Single Sign On service by using the IBM IDP:

{
    "lastName": ["SMITH"],
    "username": "http:\/\/www.ibm.com\/johnsmith@somewhere.com",
    "userUniqueID": ["http:\/\/www.ibm.com\/110000Z99Z"],
    "AUTHENTICATION_LEVEL": "2",
    "email": ["johnsmith@somewhere.com"],
    "name": ["JOHN SMITH"],
    "AZN_CRED_CREATE_TIME": "2014-09-24T02:39:13Z",
    "userRealm": "www.ibm.com",
    "userDisplayName": ["johnsmith@somewhere.com"],
    "firstName": ["JOHN"],
    "requestedAuthnPolicy": "http:\/\/www.ibm.com\/idaas\/authnpolicy\/basic\/verified_email",
    "idaas.verified_email": ["johnsmith@somewhere.com"]
}

The following table describes the profile attributes.

Table 3. Profile attributes
Attribute Description
lastName An array that typically contains 0 or 1 values. Values come from a "last name" attribute as provided in a profile that is read from the real IDP that the user logged in from. This attribute is not verified by the Single Sign On service. Use this attribute for display purposes only.
firstName An array that typically contains 0 or 1 values. Values come from a "first name" attribute, as provided in a profile that is read from the real IDP that the user logged in from. This attribute is not verified by the Single Sign On service. Use this attribute for display purposes only.
username Consists of some static text, and 2 variable portions, following the pattern:

http://<realm_identifier>/<user_identifier>

The <realm_identifier> is the same as userRealm, and represents the IDP where the user logged in.

The <user_identifier> is an identifier of the user that is provided by the real IDP where the user logged in. The identifier is expected to be unique for each user within that IDP.

For IDPs other than IBM, the username attribute is the same as the userUniqueID. For IBM, the <user_identifier> portion is the user's userDisplayName, which is unique for all IBM accounts.

Do not use this attribute for indexing user-specific data at your consuming application. Instead, use the userUniqueID.
name An array that typically contains 0 or 1 values. Values come from a "name" attribute as provided in a profile that is read from the real IDP that the user logged in from. This attribute is not verified by the Single Sign On service. Use this attribute for display purposes only.
userDisplayName An array that typically contains 1 value and is the username that the user normally sees at the IDP itself where they logged in. For most social providers, this value is also their email address. For the IBM provider, this value is the login username. This attribute is not verified by the Single Sign On service. Use this attribute for display purposes only.
userRealm One of the following values, which represent the IDP where the user logged in:
www.ibm.com
www.google.com
www.facebook.com
www.linkedin.com
userUniqueID An array that contains 1 value. This value contains some static text, and 2 variable portions, following the pattern:

http://<realm_identifier>/<user_identifier>

The <realm_identifier> is the same as userRealm, and represents the IDP that the user logged in from.

The <user_identifier> is a unique, non-recycled identifier of the user, provided by the real IDP where the user logged in.

If you store user-specific data at your own application, this attribute's value is suitable for use as an index for that data.
email An array that typically contains 1 or more values. The values come directly from a profile that was read from the real IDP where the user logged in. This attribute is not verified by the Single Sign On service. If you choose to use this attribute, verify the values in your own application.
idaas.verified_email This attribute is only available if the user has, at some time, interacted with the Single Sign On Service to complete independent email address verification using a standard email/nonce verification process. For example, if your application uses the http://www.ibm.com/idaas/authnpolicy/basic/verified_email authentication policy, then this attribute is returned. When the Single Sign On service verifies email address information, it typically starts with the value of the "email" attribute from the authenticating IDP and verifies through use of a one-time-password email.
requestedAuthnPolicy This attribute specifies the authentication policy that Single Sign On service enforces when it authenticates the user, if the consuming application requests an authentication policy. This attribute is important because when your application initially redirects to the Single Sign On service's /authorize endpoint, a malicious user might change the requested policy in the redirect. If your application requests an authentication policy, ensure that you verify that this attribute is returned in the profile, and represents the authentication policy your application originally asked for. If the attribute is missing or does not match what you requested, do not create a session for the user. Instead, redirect for authorization again or display an error.
AUTHENTICATION_LEVEL Represents the authentication level of the user credential at the Single Sign On service. Supported levels:
  • 1 - Used only at the Single Sign On service for "identity that is known, but not fully authenticated" purposes, particularly during reauthentication operations. This level is never presented to a consuming application.
  • 2 - User was authenticated by the regular process at the IDP represented by the userRealm.
  • 3 - User completed one-time-password authentication on this session.
  • 4 - User was prompted for optional browser registration. This action is typical for the "standard" authentication policy when the user is not using a previously registered browser. This level does not mean that the user necessarily registered their browser, only that they were given the option to.
AZN_CRED_CREATE_TIME Represents a time stamp of when the user most recently authenticated or reauthenticated to the Single Sign On service. You can use the value to determine authentication freshness for implementing application reauthentication policies.

User self service for Single Sign On

The Single Sign On capability provides interfaces that your users can access to manage configuration and information that is specific to their use of the application. This user-directed management is called user self service for Single Sign On.

The Single Sign On capability provides a self service home page. From this page, users can access each of the user self service actions. Optionally, you can also point users directly to each web page.

Table 4. User self service interfaces
Interface URL Description
Self service home https://idaas.ng.bluemix.net/idaas/index.jsp Entry point for user self service operations.
Manage OAuth consent decisions https://idaas.ng.bluemix.net/idaas/protected/oauthtrustedclients.jsp Allows a user to clear the settings for persisted OAuth consent decisions. On subsequent uses of an application, the user is re-prompted for consent.
View profile attributes https://idaas.ng.bluemix.net/idaas/protected/viewprofile.jsp Allows a user to view (read-only) attributes known to the Single Sign On service about the user. These are a combination of attributes from the identity provider the user authenticated with, and verified attributes managed by the Single Sign On service.
Manage devices https://idaas.ng.bluemix.net/idaas/protected/managedevices.jsp Allows a user to view and remove registered browsers. This is applicable for users of applications using the http://www.ibm.com/idaas/authnpolicy/standard authentication policy.
